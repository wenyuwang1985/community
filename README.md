# 社区生活 App 产品与技术规划方案

> 版本：v0.3 | 定位：街镇级社区连接平台 | 目标：让社区从原子化走向有机组织

---

## 一、产品定位与核心理念

### 核心价值主张

> "让住在同一片土地上的人，真正认识彼此。"

本产品不是通用社交平台的缩小版，而是以**地缘关系**为纽带的社区基础设施：

- **聊天**：邻居间沟通的替代方案（对比陌生化的业主微信群）
- **广场**：社区公共信息的流通渠道
- **集市**：熟人之间的信任交易（比闲鱼更放心）
- **互助/招工**：本地化的劳动力调度（装修、接送、搬家）

### 社区范围定义：街镇级（正科级行政单位）

采用**订阅制**多社区模型，内容严格隔离：

- 用户订阅某个街镇，即可查看该街镇的内容
- 不同社区的内容**不混合展示**，必须主动切换才能查看另一个社区
- 内容天然隔离在所属社区内，不存在跨区稀释的问题

**社区范围选择逻辑：**

| 颗粒度 | 人口规模 | 问题 |
|---|---|---|
| 小区 / 村 | 数百~数千人 | 内容太稀疏，冷启动极难 |
| **街道办 / 乡镇（推荐）** | **3~10 万人** | **信息密度与邻里感的平衡点** |
| 县 / 区 | 数十万人 | 邻里感消失，变成本地门户网站 |

**订阅机制设计：**

- 默认订阅当前 GPS 所在街镇（需授权定位）
- 用户可搜索并订阅其他街镇（故乡、旅行目的地、异地打工所在地）
- 首页顶部切换已订阅的社区（类似天气 App 城市切换），当前只看一个社区的内容
- 发帖时选择归属社区（只能选自己已订阅的）

这个设计特别适合中国大陆大量异地打工、出差人群——他们可以同时关注现居地和故乡的社区动态，互不干扰。

## 二、产品架构与页面结构

### 导航架构（路由树）

```
root
├── root_聊天
│   ├── 聊天_社区频道      每个订阅街镇自动生成公共频道（类 Discord）
│   ├── 聊天_圈子群聊      用户在本社区内自建兴趣群（广场舞、家长群等）
│   ├── 聊天_私信         一对一私信
│   └── 聊天_通知         评论、点赞、任务状态等系统消息
│
├── root_广场
│   ├── 广场_动态         社区 Feed 流（图文帖）
│   ├── 广场_集市         二手交易列表
│   ├── 广场_互助         求助 / 悬赏任务
│   └── 广场_招工         雇工 / 求职（装修、家政、临时工）
│
└── root_我的
    ├── 我的_主页         发布历史、信用评分、获得的评价
    ├── 我的_社区管理      订阅的街镇列表、切换主社区
    ├── 我的_钱包         （二期）押金、交易记录
    └── 我的_设置         通知、隐私、账号
```

### 各模块核心功能

#### 🗨️ 聊天模块

社区聊天与微信业主群的本质区别：新用户加入后能查看**历史消息**，信息不会随时间消失。

| 功能 | 说明 |
|---|---|
| 社区频道 | 每个街镇自动生成一个公共频道，所有订阅居民默认加入 |
| 圈子群聊 | 用户在本社区内自建兴趣群，需实名，群主可设置条件 |
| 私信 | 点击任何用户头像即可发起，交易时自动创建对话 |
| 消息通知 | 帖子评论、私信、任务状态变更聚合提醒 |

#### 📢 广场_动态

- 瀑布流卡片（图文混排），支持纯文字短帖
- 发帖时打 Tag：求助 / 分享 / 通知 / 问答
- 点赞、评论
- 置顶：社区管理员可置顶重要公告

#### 🛒 广场_集市

- 商品卡片：封面图 + 价格 + 新旧程度 + 发布时间
- 分类筛选：家电、家具、书籍、母婴、运动、其他
- 一键私信卖家，已售标记，支持下架

#### 🤝 广场_互助

- 发布"求助单"：描述 + 截止时间 + 悬赏积分（可选）
- 类型标签：跑腿 / 技术求助 / 信息求助 / 邻里关怀
- 接单 → 完成 → 双向评价闭环

#### 🔨 广场_招工

- 雇主发布：工种 + 工期 + 报酬 + 联系方式
- 求职者发布个人技能卡片（水电工、油漆工、保姆等）
- 完成记录 + 雇主评价 → 形成可信劳动力档案

## 三、前端技术选型

### 推荐方案：uni-app + Vue 3 + JavaScript

TypeScript 是可选项，**第一期直接用 JavaScript**，减少学习负担，先把业务跑通，待项目稳定后再考虑迁移。

**选择 uni-app 的理由：**

1. **微信小程序是最优冷启动载体**：无需上架审核，打印二维码贴公告栏即可推广，扫码即用
2. **一套代码覆盖多端**：

   | 平台 | 支持情况 |
   |---|---|
   | 微信小程序 | ✅ 原生一等支持 |
   | H5 / PC 浏览器 | ✅ |
   | iOS App | ✅（需 Apple 开发者账号） |
   | 安卓 App | ✅ |
   | 鸿蒙 Next | ✅ 官方已支持 |

3. **中国生态最完整**：微信登录、地图等 SDK 接入文档齐全

### UI 组件库

**uView UI 3.x**（免费开源），组件覆盖全面，适配小程序和 H5。

### 即时通讯方案

IM 是有状态长连接服务，不适合第一期自研，推荐接入成熟 SDK：

| 方案 | 免费额度 | 建议 |
|---|---|---|
| **腾讯云 IM（TIMSDK）** | 100 DAU 免费，功能完整 | **第一期强烈推荐** |
| 环信 | 免费版 100 人同时在线 | 备选 |
| 自研 WebSocket | 无限制，成本为服务器资源 | 第二期用户规模上来后考虑 |

## 四、后端技术选型

### 技术栈

```
Go + Gin 框架
    ├── PostgreSQL（主业务数据库）
    │       └── PostGIS 扩展（行政区划边界数据查询）
    ├── Redis（缓存 + 限流 + Session）
    ├── Cloudflare R2（图片/文件存储，出流量免费）
    └── JWT（无状态身份认证）
```

### 核心数据模型

```sql
-- 街镇表（从国家行政区划数据导入）
communities (id, name, province, city, district, level, created_at)

-- 用户表
users (id, phone, nickname, avatar, credit_score, created_at)

-- 用户订阅关系
user_community_subscriptions (user_id, community_id, is_primary, created_at)

-- 帖子表
posts (id, user_id, community_id, tag, content, images[], status, created_at)

-- 任务/互助表
tasks (id, poster_id, community_id, category, description, reward, status, deadline)

-- 商品表（集市）
items (id, seller_id, community_id, title, price, condition, images[], status)

-- 招工表
jobs (id, poster_id, community_id, job_type, duration, pay, contact, status)

-- 评价表
reviews (id, reviewer_id, reviewee_id, ref_type, ref_id, score, content)
```

### 行政区划数据来源

民政部公开数据 + 高德地图行政区划 API（免费额度充裕），初期可手动录入目标城市街镇，无需全量导入。

## 五、部署方案（零成本优先）

### 第一期目标：¥0 运营成本

**核心方案：Oracle Cloud 永久免费层 + Cloudflare 免费层**

**Oracle Cloud 永久免费层包含：**
- ARM 实例：**4 核 CPU + 24GB 内存**（跑完整后端栈绰绰有余）
- 200GB 存储、每月 10TB 出站流量

**完整部署架构：**

```
用户（微信小程序 / H5）
        ↓
Cloudflare 免费层
  ├── DNS 解析
  ├── SSL 证书（自动签发，无需手动续签）
  ├── CDN 加速静态资源
  └── DDoS 防护
        ↓
Cloudflare Tunnel（免费，无需开放服务器端口，更安全）
        ↓
Oracle Cloud ARM 服务器（永久免费）
  └── Docker Compose
      ├── Go API 服务（Gin）
      ├── PostgreSQL + PostGIS
      └── Redis
        ↓
Cloudflare R2（图片存储）
  ├── 每月 10GB 免费
  └── 出流量完全免费
```

### 关于 Let's Encrypt

不需要单独配置。Cloudflare 提供免费 SSL 证书，自动续签，已完整覆盖 HTTPS 需求。服务器与 Cloudflare 之间用 Cloudflare Origin CA 证书（永久有效）。

### 关于国内 CDN

国内 CDN（腾讯云/阿里云）需要 **ICP 备案**，备案需绑定国内服务器。第一期服务器在境外时暂时跳过。微信小程序走微信自身网络层，对境外服务器延迟不敏感，第一期完全够用。

### 各方案成本对比

| 方案 | 月成本 | 中国大陆访问 | 适用阶段 |
|---|---|---|---|
| **Oracle Cloud 永久免费** | **¥0** | 一般（Cloudflare 加速后可接受） | **第一期** |
| Hetzner VPS | ¥25~40 | 一般 | 第一期备选 |
| 腾讯云/阿里云轻量服务器 | ¥50~100 | ✅ 快速稳定 | 第二期扩张后 |

## 六、本地开发与上线流程

### 推荐开发顺序

单人开发，务必**串行推进**，不要前后端并行，节省来回对齐的成本。

```
第一步：设计数据库表结构
         在纸上或文档中把核心表想清楚再建，改结构的代价很高
         ↓
第二步：本地搭建开发环境
         Docker Compose 一键启动 PostgreSQL + Redis
         ↓
第三步：开发后端 API（Go + Gin）
         用 Apifox / Postman 测试每个接口
         一个功能模块写完测通，再写下一个
         ↓
第四步：开发前端（uni-app）
         直接对接真实 API，不用 mock 数据
         ↓
第五步：本地跑通完整闭环
         注册 → 选择社区 → 发帖 → 看帖 → 私信
         ↓
第六步：上线 Oracle Cloud
         Docker Compose 部署，Cloudflare Tunnel 接入
```

### 本地开发环境快速启动

```yaml
# docker-compose.yml（本地开发用）
version: '3.8'
services:
  postgres:
    image: postgis/postgis:16-3.4
    environment:
      POSTGRES_DB: community
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: dev123
    ports:
      - "5432:5432"

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

## 七、MVP 开发优先级

### 第一期（冷启动，目标 2~3 个月完成）

**先跑通一个街镇，验证核心闭环**

```
✅ 微信小程序登录（手机号一键授权）
✅ 街镇订阅（搜索 + 选择 + 切换社区）
✅ 广场_动态（发帖 + Feed 流 + 评论 + 点赞）
✅ 广场_集市（发布商品 + 列表浏览 + 私信卖家）
✅ 聊天_私信（接入腾讯云 IM SDK）
✅ 聊天_社区频道（公共大频道，历史消息可查）
✅ 个人主页（发布历史、基本信息）
```

**暂缓（等社区有基础活跃度后再上）：**

- 互助 / 招工模块（业务逻辑较复杂）
- 积分 / 信用体系（先积累用户数据）
- iOS / 安卓独立 App（小程序跑通后打包）

### 第二期（活跃度建立后）

```
○ 广场_互助（求助 + 接单 + 双向评价）
○ 广场_招工（劳动力市场 + 信用档案）
○ 积分激励系统（鼓励高质量内容和互助行为）
○ 多社区运营工具（管理员后台）
○ 打包发布安卓 App
○ 迁移至国内服务器 + ICP 备案 + 国内 CDN
```

## 八、冷启动运营策略

### 选点策略

- 选择**自己最熟悉的一个街镇**，宁可一个街镇做到 DAU 过百，再复制扩张
- 优先切入：老旧社区（业主群乱、物业差）或大型新建社区（邻居互不相识）

### 线下推广

1. **公告栏二维码**：打印 A4 贴在社区入口、电梯间、快递驿站
2. **物业合作**：帮物业发公告，物业有传播动力
3. **关键节点渗透**：菜市场摊主、便民超市老板——他们是社区信息枢纽
4. **创始居民标识**：早期用户永久显示"XX 号创始居民"，激励传播

### 内容冷启动

- 初期**自己多发帖**：二手出售、本地资讯、生活分享，制造内容氛围
- 将现有微信群中有价值的信息引导过来
- 设置每日话题激活讨论

## 九、技术风险与应对

| 风险 | 应对策略 |
|---|---|
| 微信小程序审核拒绝 | 提前准备材料；初期用 H5 版本兜底 |
| 境外服务器访问慢 | Cloudflare Tunnel + 边缘缓存；第二期迁移国内 |
| IM 消息量超腾讯免费额度 | 监控 DAU，超过 100 前评估升级或自研 |
| Go 后端经验不足 | 模块化开发，AI 辅助；优先跑通业务，优化在后 |
| 社区内容违规 | 举报机制 + 关键词过滤，初期人工审核 |

## 十、技术选型总结

| 层级 | 选型 | 理由 |
|---|---|---|
| 前端框架 | uni-app + Vue 3 + **JavaScript** | 一套代码覆盖小程序/H5/App，JS 上手快，无 TS 负担 |
| UI 组件 | uView UI 3.x | 免费开源，组件完整 |
| 即时通讯 | 腾讯云 IM SDK | 省去自研 IM 工作量，免费额度够用 |
| 后端语言 | Go + Gin | 高并发，语法简洁，AI 辅助效果好 |
| 主数据库 | PostgreSQL + PostGIS | 行政区划查询支持，开源免费 |
| 缓存 | Redis | 会话管理、限流、热点数据缓存 |
| 文件存储 | Cloudflare R2 | 出流量免费，S3 兼容 API |
| 服务器 | Oracle Cloud ARM（永久免费） | 4核24G，足够支撑早期用户 |
| 网关/CDN | Cloudflare 免费层 | DNS + SSL + CDN + Tunnel 全覆盖 |
| 部署方式 | Docker Compose | 简单可控，适合单人运维 |
| 小程序登录 | 微信手机号授权 | 门槛低，覆盖广，身份可信 |
